# PPT Creator Skill Pan

## Purpose

把源材料制作成“高质量、主要信息可编辑、后期可维护”的 PPT / PPTX。默认工作流不是“先整页生图，再从图片里抢救可编辑性”，而是：

1. 内容与数据底稿
2. 页面实现合同
3. 视觉资产规格
4. 原生可编辑 PPT 搭建
5. 高保真修饰与验收

这套 skill 不只适用于论文科研，也适用于企业汇报、方案汇报、行业报告、经营分析、培训材料、产品介绍、研究摘要等文档转 PPT 场景。

## Use This Skill When

- 用户要把 PDF、Word、报告、论文、截图集合、网页摘录或混合材料制作成 PPT
- 用户要求主要内容可编辑，而不是整页图片化
- 用户希望先把内容、结构、数据理清，再做设计
- 用户需要较稳定、较自动化的 PPT 生产流程

## Do Not Default To This Skill When

- 用户只想先看几张纯视觉概念图
- 用户要的是海报、封面插画、KV，而不是可编辑 PPT
- 用户明确接受绝大多数页面直接做成整页图片

## Tool Roles

- `presentations`：负责 PPT 原生对象层，包括文本框、形状、表格、原生图表、版式骨架、页面结构与最终 `pptx`
- `spreadsheets`：负责数据语义层，包括字段整理、公式、指标口径、数据表、图表数据源、表格底稿与数据一致性检查
- `imagegen / Image2`：只负责局部视觉资产层，包括封面视觉、章节页背景、局部插画、复杂纹理、照片合成、少量复杂示意资产；不负责默认整页成稿

一句话分工：

- `presentations` 管“页”
- `spreadsheets` 管“数”
- `imagegen` 管“局部视觉资产”

## Compatibility

最佳使用环境：

- 当前 Codex 运行环境同时具备 `presentations`、`spreadsheets`、`imagegen / Image2`

兼容性说明：

- 如果缺少 `presentations`，这个 skill 仍可用于前 3 个阶段的分析、规划和视觉资产规格制定，但无法稳定完成原生可编辑 `pptx` 交付
- 如果缺少 `spreadsheets`，这个 skill 仍可制作非强数据型 PPT，但数据表、公式、指标口径、原生图表数据源的稳定性会下降
- 如果缺少 `imagegen`，这个 skill 仍可完成主要信息可编辑的 PPT，只是局部视觉资产的丰富度和风格表现会受限
- 因此，别人下载后一般可以使用这套工作流，但若要完整发挥效果，目标环境最好具备以上三类能力

## Confirmation Policy

默认采用自动推进模式：

- 阶段 1、2、4、5 在没有高风险歧义时自动继续执行
- 只有阶段 3 的风格 / 视觉资产方向需要用户明确选择一次
- 如果用户事先明确授权自动选风格，则阶段 3 也可自动继续

只有在以下异常下，才允许中途打断并请求确认：

- 高仿真参照物缺失，但用户坚持要求高仿真
- 数据明显不完整，只能做视觉近似，且会影响结论表达
- 机制图、流程图、结构示意图存在无法从源材料确认的关键关系
- 最终真实打开验证出现 repair / 修复提示或严重兼容性问题

## Non-Negotiable Rules

1. 不要把默认方案做成“整页图片 + 少量文字覆盖”
2. 不要为了高仿真牺牲主要信息可编辑性
3. 不要为了可编辑而把复杂视觉粗糙重建成低质默认组件
4. 数据表页、原生图表页、常规正文页、双栏信息页、对比页、时间线页、结论文字页，默认禁止整页交给 `imagegen`
5. 除非用户明确接受“视觉近似”或“图片保留”，否则正文信息层、数字层、图表标签层、表格文字层优先原生实现
6. 若无明确参考 deck / 模板 / 样张 / 品牌规范，不得虚构“高仿真完成”；只能表述为高质量一致风格
7. 所有逐页与逐资产信息都要复用稳定 ID：页面用 `page_id`，数据表用 `data_table_id`，视觉资产用 `asset_id`
8. 未经数据底稿证实的数字、比例、时间序列和图表数据，不得伪装成正式权威原生图表
9. 机制图、流程图、结构示意图等必须核对节点、箭头、层级、术语和图例含义，不能只看起来“像对”
10. 最终交付前必须进行一次真实打开验证，至少在 PowerPoint 或 WPS 其中一种中实际打开检查

## Operating Workflow

### Stage 1: Build content, data, and fidelity grounding

目标：

- 读完用户提供的全部源材料
- 搭建后续 PPT 所需的内容底稿、数据底稿和视觉参照约束
- 判断是“有明确高仿真参考”还是“只能做高质量原创风格”

必须产出：

- 推荐页数与推荐叙事顺序
- 逐页清单：`page_id`、页面标题、页面目标、核心观点、正文要点、`page_type`
- 每页的 `editability_level`，只能在“高可编辑”“中等可编辑”“允许局部图片保真”中选一档
- 对涉及数据、表格、图表的页面给出数据底稿要求：`data_table_id`、字段名称、字段含义、指标口径、单位、时间维度、分组维度、来源、是否需要公式、适合表格还是图表
- 页面实现策略摘要：逐页标注更偏 `presentations`、`spreadsheets + presentations`、还是 `imagegen + presentations`
- 禁止图片化页面清单
- 字体与替代链清单

规则：

- 当前阶段只做分析与结构化
- 不要生成 PPTX
- 不要生成整页图片
- 不要直接调用 `imagegen` 生成页面样张
- 信息不足时标记“需确认”，不要编造

### Stage 2: Convert planning into an implementation contract

目标：

- 把阶段 1 的规划落成可执行的页面实现合同
- 把数据真正前置到可执行的数据底稿，而不是拖到搭 PPT 时才第一次处理

必须产出：

- 逐页实现合同：页面结构分区、必须原生可编辑的元素、允许作为图片保留的元素、禁止整页图片化的原因
- 数据实现方案：每个数据页使用哪个 `data_table_id`、需要哪些字段、是否有公式、是否有派生指标、图表是否原生构建
- 数据实体化交付清单
- 全局设计系统合同：标题层级、正文层级、数字层级、图表字体层级、颜色体系、背景策略、卡片样式、边框/分隔线语言、页脚页码规则、图片裁切原则、留白原则、字体替代链表

规则：

- 复杂机制页、流程页、矩阵页、时间线页，默认先由 `presentations` 定义原生骨架
- 不得默认把这些页面整页交给 `imagegen`
- 这一步仍然不要生成最终 PPTX

### Stage 3: Generate only reusable visual asset specifications

目标：

- 只为确有必要的页面设计视觉资产规格
- 产物必须是局部资产方案，而不是整页成品替代品

必须产出：

- 2 到 3 套视觉方向方案
- 推荐方案
- 视觉资产交付清单：每个资产包含 `asset_id`、名称、对应 `page_id`、用途、尺寸/比例、是否透明背景、裁切方式、是否复用、文字禁入区、与原生对象叠加关系、降级方案

如果资产来自 PDF、网页、报告、截图、论文、海报、产品界面或其他源材料截取，必须额外写明：

- 保留区域
- 应去除的污染区域
- 不可误删的有效信息
- 是否需要人工感知层面的二次核对

规则：

- 禁止输出数据表页、原生图表页、常规正文页、双栏信息页的整页终稿图片
- 禁止把典型正文页做成后续临摹母版
- 不要新增未经确认的数据、品牌、Logo、人物、产品或来源
- 如果资产服务于机制图、流程图、结构示意图或复杂信息图，要同步输出语义核对要点
- 对于源材料截取类资产，执行“去污染但不过裁”原则：去掉无关正文、相邻栏、页眉页脚、页码、边缘残片和误入文字，但不得误删标题、坐标轴、图例、标签、注释、必要留白和语义必需区域

确认：

- 默认只在这个阶段询问一次风格 / 视觉资产方向

### Stage 4: Ground data first, then build the editable deck natively

目标：

- 正式输出主要信息层可编辑、视觉完成度高、后期可维护的 16:9 `pptx`

执行顺序：

1. 用 `spreadsheets` 落地数据字典、指标计算、图表数据表、表格底稿、来源说明
2. 用 `presentations` 搭建标题层、正文层、形状层、表格层、原生图表层与页面结构层
3. 只在必要页面叠加阶段 3 确认过的局部视觉资产

关键规则：

- 标题、副标题、正文、页码、页脚、关键词、数字卡片、结论框、图表标签、图例、表格文字、注释等主要信息层必须优先使用原生可编辑对象
- 数据表页、原生图表页、常规正文页、双栏信息页、对比页、时间线页默认禁止整页图片化
- 流程页、矩阵页、机制页默认先用原生形状和文本构建逻辑骨架，再视需要叠加局部视觉资产
- 所有文字块、项目符号块、标题块、数字卡片与说明文字，必须完全落在所属容器、底板、色块或安全区内，不得压线、越界、悬空贴边或与背景框明显错位
- 若字体替代、字号变化或自动换行导致容器失配，必须先回调容器尺寸、内边距或文本框位置，再继续
- 对缺少真实数据支撑的图表，只能明确标注为“视觉近似图表”或“示意性图表”

必须同步产出：

- 渲染预览
- 页面可编辑性清单
- 数据落地清单
- 字体落实清单
- 机制图 / 示意图语义核对清单

### Stage 5: Final fidelity polish and hard acceptance

目标：

- 在不破坏主要信息可编辑性的前提下，做最后一轮高保真修饰与验收

必须检查：

- 标题层级、文字换行、字号、字重、对齐、留白、颜色、模块比例、页脚页码、图表标签、表格边界与资产叠加关系
- 图表、表格、数字页与数据底稿的一致性
- 字体集、替代链、换行结果、预览结果的一致性
- 流程图、机制图、结构示意图和复杂信息图的语义正确性
- 真实打开验证：repair / 修复提示、不可读内容删除、对象丢失、字体替换、版式漂移
- 所有容器类版式是否存在文字越界、压线、贴边或背景框错位
- 所有源材料截取类资产是否既去除了污染区域，又保留了标题、图例、坐标轴、标签、注释和必要留白，不得因净化而过裁

最终验收至少确认：

- PPTX 为 16:9
- 页序正确
- 主要标题、正文、关键词、数字、页码、图例、表格文字、图表标签可编辑
- 禁止整页图片化的页型没有被整页图片化
- 没有明显错位、乱码、重影、溢出、缺字、错误页码或比例异常
- 数据口径一致
- 复杂视觉资产使用范围合理
- 视觉近似图表已被明确标注
- 高仿真依据的参照物已说明，若无参照物则明确表述为高质量一致风格终稿

## Output Contract

最终交付时应尽量同时提供：

- 最终 `pptx`
- 预览图或渲染预览
- 简洁交付说明，至少包含：
  - 属于高仿真终稿还是高质量原创终稿
  - 高仿真参照物是什么
  - 哪些内容可编辑
  - 哪些内容是局部视觉资产
  - 哪些数据或图表为视觉近似
  - 哪些字体用了替代链
  - 真实打开验证结果如何
  - 还存在哪些限制

## Quality Bar

这个 skill 成功的标准不是“做出几张看起来像 PPT 的图片”，而是：

- 先把内容与数据打牢
- 大多数页面主要信息层可编辑
- `presentations`、`spreadsheets`、`imagegen` 的分工稳定
- 风格统一且细节成熟
- 最终文件真实可打开、可维护、可继续改稿
